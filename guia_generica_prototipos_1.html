01 · Qué es

¿Qué es un prototipo HTML?

Un prototipo HTML es una propuesta UX navegable — el cliente puede hacer click, completar formularios y sentir el flujo real. No es una imagen estática ni un video, es una mini aplicación que vive en un solo archivo.

🏗

HTML — Estructura

Define qué elementos existen: botones, formularios, tablas, paneles, navegación lateral.

🎨

CSS — Visual

Define cómo se ven: colores de marca, tipografía, espaciados, bordes, estados activo/inactivo.

⚡

JavaScript — Interactividad

Define cómo se comporta: clicks, navegación entre pasos, búsqueda, totales dinámicos.

💡

La clave: todo vive en un solo archivo .html. Se abre en cualquier browser, sin instalar nada, sin servidor. Puedes enviarlo por email, subirlo a GitHub o abrirlo desde el escritorio.

02 · Cuándo usarlo

¿Cuándo usar un prototipo HTML?

No siempre es la herramienta correcta. Úsalo cuando necesites mostrar interacción real, no solo diseño visual.

✅ Ideal para esto

  • Flujos multi-paso con navegación entre pantallas
  • Formularios con validación y lógica condicional
  • Propuestas donde el cliente necesita "sentir" el flujo
  • Demos con datos reales del sistema legacy
  • Cuando Figma es muy estático para explicar la UX
  • Presentaciones donde el cliente interactúa en vivo
  • Comparativos before/after (sistema actual vs propuesta)

❌ No es ideal para esto

  • Diseño de componentes aislados (mejor Figma)
  • Pantallas con animaciones complejas
  • Prototipos que necesitan datos en tiempo real de una API
  • Cuando el cliente solo necesita ver imágenes estáticas
  • Flujos con decenas de pantallas diferentes
  • Cuando el equipo ya tiene Figma con interactividad

04 · Proceso

Proceso paso a paso

Sigue esta secuencia. No saltes pasos — es tentador ir directo al prototipo completo, pero construir y validar de a un paso a la vez ahorra mucho tiempo.

1

Define el flujo completo

Antes de que Claude escriba una sola línea de código, define cuántos pasos tiene el flujo, cómo se llama cada uno y qué hace el usuario. Si tienes un brief o documento, compártelo directamente.

Prompt de ejemplo
"Quiero crear un prototipo HTML para el flujo de [nombre del flujo]. Tiene [N] pasos: Paso 1: [nombre] — [qué hace el usuario] Paso 2: [nombre] — [qué hace el usuario] Paso 3: [nombre] — [qué hace el usuario] Te adjunto el documento de referencia y screenshots del sistema actual."
2

Comparte el design system

Adjunta screenshots del sistema nuevo (si existe) o del sistema actual para que Claude entienda el look & feel que debe replicar. Cuantas más pantallas compartas, mejor entiende Claude los colores, tipografía, componentes y espaciados.

Prompt de ejemplo
"Aquí van los screenshots del design system. El sistema se llama [nombre]. Los colores principales son [color primario] y [color secundario]. Quiero que el prototipo replique este estilo exactamente: - Header: [descripción] - Sidenav lateral con los pasos del flujo - Botones: [descripción]"
3

Construye el Paso 1 y valídalo

Pide que Claude construya solo el primer paso. Abre el archivo en Chrome, revísalo visualmente y pide todos los ajustes que necesites antes de continuar. Es mucho más fácil ajustar una pantalla que cuatro.

💡 Tip de validación Cuando abras el archivo, revisa: ¿Se ven bien los espaciados? ¿Los colores son correctos? ¿Los botones tienen el tamaño adecuado? ¿El texto es legible? Pide ajustes específicos con capturas de pantalla como referencia.
4

Construye el resto de pasos uno a uno

Una vez validado el Paso 1, construye el Paso 2, valídalo, luego el Paso 3, y así. Cada paso hereda el design system del anterior, por lo que los ajustes globales (colores, tipografía) se aplican a todo.

Prompt de ejemplo
"El paso 1 se ve bien. Ahora construye el Paso 2 — [nombre]. Debe incluir: - [elemento 1] - [elemento 2] - [elemento 3] Cuando el usuario hace [acción], debe pasar [resultado]."
5

Integra todo en un solo archivo

Una vez listos todos los pasos, pide la integración en un único HTML. Este paso es crítico — evita problemas de navegación entre archivos separados y garantiza que el CSS sea consistente en todas las pantallas.

Prompt de ejemplo
"Integra todos los [N] pasos en un solo archivo HTML. La navegación entre pasos debe funcionar con los botones de cada pantalla. El sidenav debe mostrar el estado de cada paso: activo (color primario), completado (checkmark), pendiente (gris). No debe haber recarga de página al navegar."
6

Agrega interactividad avanzada

Con el flujo básico funcionando, agrega los comportamientos interactivos que hacen el prototipo más realista: búsquedas, filtros, totales dinámicos, modals de confirmación, selección de opciones.

💡 Prioriza la interactividad que el cliente verá No necesitas que todo funcione. Enfócate en los momentos clave que quieres mostrar en la demo: el search, el formulario principal, la confirmación final.
7

Publica y comparte

Sube el archivo a GitHub Pages para obtener un link público que el cliente puede abrir en cualquier browser sin descargar nada. También puedes tomar screenshots de cada estado para Figma o presentaciones.

Siempre incluido
⚡

07 · Ajustes

Cómo pedir ajustes a Claude

La clave es ser específico. Claude no puede ver tu pantalla — descríbele exactamente qué está mal y adjunta screenshots cuando sea posible.

Tipo de ajusteCómo pedirlo
Espaciado / márgenes"El margen izquierdo del menú debe ser 48px, igual que el contenido del lado derecho"
Colores"El botón Confirmar debe ser rojo (#e31837), no verde. El texto de los pasos inactivos en gris (#aaa)"
Tamaño de elementos"Agranda el sidebar de resumen — actualmente tiene 224px, ponlo en 320px"
Texto / labels"Renombra 'Items' a 'Create Order' en el sidenav, en el título de la página y en el bottom bar"
Estructura / layout"Mueve los botones Cancel/Back/Next al lado derecho del título, no en la barra inferior"
Funcionalidad nueva"Al hacer click en Search Order debe aparecer una animación de 'Searching...' por 0.8 segundos antes de mostrar los resultados"
Algo que no funciona"El ícono del sidenav no cambia a blanco cuando el paso está activo. El problema es que el SVG usa stroke heredado"
Replicar algo visual"Quiero que se vea exactamente como en este screenshot que te adjunto" [+ imagen]
08 · Buenas prácticas

Buenas prácticas

✅ Haz esto

  • Valida cada paso visualmente antes de pasar al siguiente
  • Adjunta imágenes cuando quieras replicar algo
  • Pide un solo ajuste a la vez — no en lista
  • Trabaja en un solo archivo desde temprano
  • Comparte el contexto del proyecto al inicio
  • Usa datos realistas en el prototipo
  • Guarda copias antes de cambios grandes
  • Toma screenshots de cada estado interactivo
  • Si cambias de conversación, sube el archivo .html al inicio

❌ Evita esto

  • Pedir 10 cambios a la vez en un solo mensaje
  • Trabajar con archivos separados por paso
  • Hacer ediciones manuales en el HTML sin avisar
  • Asumir que Claude recuerda conversaciones anteriores
  • Usar Google Drive para compartir (no renderiza HTML)
  • Esperar que Figma vía código quede pixel-perfect
  • Agregar interactividad compleja antes de validar el diseño base
  • Saltarte la validación visual entre pasos
09 · Publicación

Cómo compartir el prototipo

MétodoVentajasDesventajasRecomendado para
GitHub Pages Gratis, link permanente, fácil de actualizar Requiere cuenta de GitHub Presentaciones a cliente
Netlify Drop Sin cuenta, link en 30 segundos Link temporal (expira) Revisiones rápidas
Archivo local Sin dependencias externas No se puede compartir por link Uso personal / demo en vivo
CodePen / JSFiddle Link compartible, no requiere cuenta Límite de tamaño, pega el código Prototipos pequeños
🚀

GitHub Pages — paso a paso: Crea un repositorio en github.com → sube el archivo como index.html → ve a Settings → Pages → selecciona Branch: main → Save. En 1-2 minutos tendrás un link tipo https://tuusuario.github.io/nombre-repo
